CODE STACK REPOSITORY AUDIT REPORT
===================================

EXECUTIVE SUMMARY
-----------------
This audit examined all source files in the Code Stack repository to determine which 
features are fully functional end-to-end versus partially implemented or missing. The 
investigation reveals that while backend logic exists for most features, the system 
suffers from a critical architectural gap: generated content requires manual user 
approval and action before any code changes are committed, PR'd, or deployed. This is 
the root cause of users feeling the system "just responds but doesn't develop the project."

ROOT CAUSE ANALYSIS: WHY USERS FEEL "NOTHING GETS DONE"
=======================================================

1. MANUAL APPROVAL BOTTLENECK
-----------------------------
The orchestrator (/api/orchestrator/route.ts) correctly classifies intents and generates 
content, but stops short of automatic execution:

- TESTS: Generated tests appear in chat AND set pendingChange state, but the user MUST 
  click "Commit" or "Create PR" buttons. If the user doesn't notice the amber pending 
  change panel, nothing happens.
  
- DOCS (README/OpenAPI): Generated content is displayed in chat but no pending change 
  panel is created. The user would need to manually copy/paste the content into a file.

- REFACTOR: Plan is shown with checkboxes. User must click "Approve & Execute" which 
  triggers an external GitHub Actions workflow (refactor.yml). If this workflow doesn't 
  exist in the target repository, execution silently fails or returns an error.


2. MISSING DIRECT FILE WRITE CAPABILITY
---------------------------------------
The system has NO mechanism to directly write files to the repository without going through:
  1. User clicks "Commit" -> calls /api/github/commit -> GitHub Contents API PUT
  2. OR user clicks "Approve & Execute" -> dispatches refactor.yml workflow -> external script runs

There is no "apply all changes automatically" option. Every code modification requires 
explicit user confirmation.


3. UI CONTROLS NOT CONNECTED TO ALL FEATURES
--------------------------------------------
Feature               | Backend Exists | Dedicated Button? | Gap
----------------------|----------------|-------------------|----------------------------------
Deploy                | YES            | NO                | No "Deploy Now" button in UI
Webhook Trigger       | YES            | NO                | No UI to trigger custom webhooks
README Commit         | YES            | NO                | Generated README shown but no save button
OpenAPI Commit        | YES            | NO                | Generated spec shown but no save button


4. ORCHESTRATOR DESIGN LIMITATION
---------------------------------
The orchestrator returns requiresApproval: true for refactor, commit, PR, and deploy 
actions. However, the frontend only shows approval controls for:
- Refactor plans (checkboxes + execute button)
- Pending changes (commit/PR buttons)

But for docs generation, there's no parallel "pending change" UI—the content is just 
displayed as text.


ORCHESTRATOR EFFECTIVENESS ASSESSMENT
=====================================

INTENT CLASSIFICATION
---------------------
WORKING WELL: The orchestrator uses Groq (llama-3.1-8b-instant by default) to classify 
user messages into 13 action types. JSON parsing includes fallback to chat if parsing fails.

MODEL IDs - CURRENT STATUS
--------------------------
UP-TO-DATE with user's specified list:
- llama-3.1-8b-instant: used for chat, architecture, search, commit messages, CI
- qwen/qwen3-32b: used for tests, audit, refactor fallback
- meta-llama/llama-4-scout-17b-16e-instruct: used for refactor (primary), docs

REPOSITORY CONTEXT PASSING
--------------------------
IMPLEMENTED BUT LIMITED:
- File lists (up to 200 files) are passed to Groq prompts for refactor, architecture, 
  docs, search, and greeting actions
- Individual file content is passed when a file is selected
- System prompt explicitly warns model not to invent file names when context is unavailable

WEAKNESS: Only file paths are included, not file contents (except for the currently 
selected file). This limits the AI's ability to make informed cross-file changes.

UI elements are needed:

----------------------|----------------------------------|----------------------------------------

SPECIFIC BLOCKERS IDENTIFIED
============================

1. TEST GENERATION FLOW (dashboard/page.tsx, lines 305-314)
----------------------------------------------------------
PROBLEM: Creates pending change but user may not notice the amber panel below chat. 
No auto-scroll to panel, no notification.


2. DOCS GENERATION (orchestrator route, lines 230-236)
------------------------------------------------------
PROBLEM: Result is returned as text only. No pendingChange is set. User must manually 
copy content.


3. REFACTOR EXECUTION DEPENDENCY (api/refactor/execute/route.ts, lines 14-37)
-----------------------------------------------------------------------------
PROBLEM: Requires refactor.yml workflow to exist in target repository. Most repos 
won't have this. No fallback to direct commit.


4. DEPLOYMENT TRIGGER (No UI Integration)
-----------------------------------------
Backend exists at /api/deploy/trigger and in actions.ts (triggerDeploy), but:
- No button in dashboard
- Only accessible via typing "deploy" in chat
- Returns requiresApproval: true but no approval UI is rendered for this case


CONCLUSION
==========

The Code Stack system is architecturally sound with comprehensive backend implementations 
for all major features. However, it operates in a "conversational assistant" mode rather 
than an "autonomous developer" mode due to:

1. MANDATORY APPROVAL GATES for all write operations
2. MISSING UI CONTROLS for deploying, saving docs, and batch operations
3. EXTERNAL WORKFLOW DEPENDENCIES (refactor.yml) that may not exist
4. PASSIVE PRESENTATION of generated artifacts (tests, docs) without prominent save mechanisms

To address user complaints that the system "just responds but doesn't develop," the 
product decision required is: Should AI changes be applied automatically by default, 
or always require explicit approval?

Currently, the system enforces the latter. Changing this behavior would require:
- Adding an "auto-commit" mode in settings
- Converting all pendingChange panels to auto-execute when enabled
- Creating direct file-write fallbacks instead of relying on external workflows
- Building prominent UI buttons for deploy, save docs, and run tests
